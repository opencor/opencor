The best way to familiarise yourself with OpenCOR's internals is by having a look at its source code.

Namespace

All of OpenCOR's code is and must be made available under the OpenCOR namespace (e.g. [OpenCOR]/src/mainwindow.cpp and [OpenCOR]/src/mainwindow.h). There are two exceptions to this rule: [OpenCOR]/src/main.cpp and [OpenCOR]/windows/src/main.cpp (i.e. OpenCOR's two main .cpp files).

Issues

All changes to OpenCOR's codebase must be referenced in the list of issues using labels. There are three types of labels:

  • Version: they are used to request a or an for a future version of OpenCOR or to report a with the current version of OpenCOR.
  • Development: they are used to specify a that needs to be carried out (in preparation of the next version of OpenCOR) or to report a issue with the current development version of OpenCOR.
  • Platform: they are used to specify the platform(s) (, and/or ) affected by a , or issue.
Continuous integration

Whenever something is pushed to our Git repository, OpenCOR gets automatically built and tested on Travis CI's Ubuntu and OS X machines. That is, unless [ci skip] has been added to a commit comment. However, and as a general rule, a commit that closes an issue should always trigger OpenCOR to be built and tested on Travis CI.
