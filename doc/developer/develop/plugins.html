Plugins

OpenCOR relies entirely on plugins. So, though it works with no plugin, it is of no use at all.

Some of the plugins provide access to third-party libraries while others are specific to the organisation or editing side of OpenCOR, or are plugins that do not fit in any of the aforementioned categories.

<pluginname> namespace

As for OpenCOR itself, a plugin's code is and should be made available under the OpenCOR namespace (e.g. [OpenCOR]/src/plugins/misc/Core/src/coreplugin.cpp and [OpenCOR]/src/plugins/misc/Core/src/coreplugin.h).

<pluginname>plugin.*

A plugin could be released as a pure shared library, but this would make it unnecessarily cumbersome to use. So, instead, we rely on Qt's support for plugins. To that end, we must create a couple of files, such as [OpenCOR]/src/plugins/misc/Core/src/coreplugin.cpp and [OpenCOR]/src/plugins/misc/Core/src/coreplugin.h for the Core plugin. A plugin may implement one or several interfaces. For example, CorePlugin implements GuiInterface which inherits from PluginInterface.

Basic information

Every plugin must provide the following information:

  • Interface version: OpenCOR may, in the future, use a different interface to communicate with plugins. It must therefore ensure that a plugin uses the same interface before loading it.
  • Type: a plugin can be used for the console version of OpenCOR and/or its GUI version, and the type of the plugin is used to determine whether the plugin can be loaded.
  • Manageable: specify whether a user can change the loading state of a plugin.
  • Dependencies: some plugins depend on other plugins to work. OpenCOR must therefore know about them, so that it can make sure that they also get loaded.
  • Description: a short description of the plugin.

This information is, for the Core plugin, provided in [OpenCOR]/src/plugins/misc/Core/src/coreplugin.cpp and [OpenCOR]/src/plugins/misc/Core/src/coreplugin.h through the PLUGININFO_FUNC CorePluginInfo() function. As can be seen, the name of the function is related to the name of the plugin and is <PluginName>PluginInfo. OpenCOR uses the name of the plugin file to determine the name of that function. This ensures that only one copy of a given plugin is loaded. Otherwise, support for internationalisation would normally be done using Qt's tr() function, but it cannot be applied in this particular case. So, instead, we use a QMap approach, as can be seen from [OpenCOR]/src/plugins/misc/Core/src/coreplugin.cpp.

Import/export

To support plugin dependencies requires some classes and/or functions to be imported/exported (using Q_DECL_IMPORT/Q_DECL_EXPORT, respectively). To handle this, we must create a header file which defines a macro that refers to either Q_DECL_IMPORT or Q_DECL_EXPORT, depending on how the plugin code is to be compiled. For the Core plugin, the header file in question is [OpenCOR]/src/plugins/misc/Core/src/coreglobal.h.

Deployment

Our CMake ADD_PLUGIN macro takes care of the deployment of a plugin. However, on Mac OS X, we must ensure that all the Qt libraries required by the plugins are embedded within the OpenCOR bundle and, more precisely, within [OpenCOR.app]/Contents/Frameworks. Indeed, macdeployqt can only deploy the libraries required by OpenCOR itself, not its plugins. So, it is up to us to determine which libraries are required and to deploy them:

  • Qt dependencies: our shell script, qtdependencies, can be used to check which libraries are required by one or several Qt binaries. Thus, to check which Qt libraries are required by OpenCOR's plugins, we could enter the following on the command line, this after having built OpenCOR: 
    $ cd [OpenCOR]
$ doc/developer/develop/res/qtdependencies build/OpenCOR.app/Contents/PlugIns
QtCore
QtDBus
QtDeclarative
QtGui
QtHelp
QtNetwork
QtOpenGL
QtScript
QtSql
QtSvg
QtWebKit
QtXml
QtXmlPatterns
phonon
    From there, we can compare the above list against the list of Qt libraries that got deployed to [OpenCOR.app]/Contents/Frameworks by macdeployqt, and see which ones are missing.
  • Qt libraries deployment: any missing Qt library can be deployed using our CMake DEPLOY_MAC_OS_X_QT_LIBRARY macro, but first we must determine the library's Qt dependencies. For example, for QtHelp, we could enter the following on the command line 
    $ [OpenCOR]/doc/developer/develop/res/qtdependencies /Library/Frameworks/QtHelp.framework/Versions/4/QtHelp
QtCore
QtGui
QtHelp
QtNetwork
QtSql
QtXml
    The QtHelp dependency can be ignored, leaving us with the following call to DEPLOY_MAC_OS_X_QT_LIBRARY which is also used in [OpenCOR]/CMakeLists.txt: 
    DEPLOY_MAC_OS_X_QT_LIBRARY(QtHelp
    QtCore
    QtGui
    QtNetwork
    QtSql
    QtXml
)
    This will embed QtHelp within the OpenCOR bundle, after having stripped it from anything that is not essential (thus reducing its size) and having ensured that its Qt dependencies (QtCore, QtGui, QtNetwork, QtSql and QtXml) point to our embedded version of the Qt libraries.
