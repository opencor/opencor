Miscellaneous

Ideas Worth Exploring/Implementing

Should any of the ideas below ever get implemented, then they should then be documented and put somewhere under the user documentation.

  • For various reasons (test suites, parameter sweep, etc.), it would be good for OpenCOR to also work as a command line tool, and have it run all the simulations (or whatever is required of it), and then exit. This might require an input file of sorts, which may or not prove practical. For example: 
    $ ./OpenCOR -i myInputFile
    Indeed, in some cases, it would probably be more convenient if one could just pass some arguments to OpenCOR to, say, change the default value of this or that parameter (rather than having to write another input file). In that context, maybe one could have some XML file that describes a particular set of command line options (together with corresponding actions, using some kind of a scripting language?) which would be passed together with the 'normal' command line options. For example, we could have something like: 
    $ ./OpenCOR -s mySyntaxFile -a -c 1 -o myOutputFile
    where mySyntaxFile would be the aforementioned XML file and the rest various flags/options. The downside with this approach is that it might be quite tricky to implement. In the end, it might be reasonable to have an input file and have model users write a script that would accept the kind of flags/options they are after, and then generate a temporary input file that would be passed to OpenCOR? With regards to the input file itself, SED-ML would be an obvious candidate, but this would not tackle the multi-cellular capabilities of OpenCOR (see below)...
  • OpenCOR is aimed at organising, editing, simulating and analysing CellML files, i.e. focussing on 'cellular' modelling. However, 'multi-cellular' modelling is something that should be of interest to many users. In fact, several COR users have already asked for such a feature. The idea would, here, be to take advantage of existing multi-cellular libraries, e.g. OpenCMISS and Chaste, i.e. in the same way that we are taking advantage of the CellML API developed by people at the Auckland Bioengineering Institute.
