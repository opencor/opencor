Known Issues/Limitations

Please note that the following known issues/limitations have been observed on Windows 7, Ubuntu 11.04 (Natty Narwhal; both 32-bit and 64-bit) and Mac OS X 10.7 (Lion). So, if you use a different operating system, these issues/limitations may or not be relevant to you.

Windows, Linux and Mac OS X

By default, OpenCOR uses your system's language for menus, dialog boxes, etc., as long as it is either English or French (please contact us if you would like another language to be added). If your system uses another language, then OpenCOR will default to English. Otherwise, if you specifically specify English or French as the language to be used by OpenCOR, then please be aware that system messages, dialog boxes, etc. will still be displayed in your system's language.

Windows

  • OpenCOR's embedded file browser may, on some computers, result in OpenCOR being slow to respond at startup. Unfortunately, this is a very well-known problem with Windows, but thankfully there are several known causes and solutions which you might find useful.
  • Resetting all the user settings (by selecting the Tools|Reset All menu) and applying plugin changes (by selecting the Tools|Plugins... menu) requires restarting OpenCOR. There may, however, be times when OpenCOR will have to be restarted manually. We believe this to be a problem with Windows itself (since this does not happen on either Linux or Mac OS X), even though we have yet to determine its exact cause.

Linux

Depending on your window manager (e.g. Compiz) and or desktop theme, OpenCOR may or not experience one of the following issues/limitations (none of which affect the good functioning of OpenCOR itself):

  • Upon starting OpenCOR, the software will try to retrieve your previous session's settings which may result in the main window's position being slightly shifted.
  • To show the help window by pressing the F1 key may, if the help window is floating (as opposed to being docked to the main window), result in OpenCOR not being the active application anymore. This means that if you were to press the F1 key one more time, then the help window would remain visible while it should have disappeared.
  • Menu items that have sub-menus may not, on some versions of Gnome, have their icon displayed (should they normally have one). This has been discussed in many places, such as here and here. Anyway, one way to change this behaviour is to enter the following on the command line: 
    $ gconftool-2 --type Boolean --set /desktop/gnome/interface/menus_have_icons True

Note that the above issues have nothing to do with OpenCOR itself (everything works fine on both Windows and Mac OS X), but with the window manager that you use. Indeed, there is a lack of compliancy between the various window managers, so something that works with one window manager may or not work with another, as is unfortunately the case here.
